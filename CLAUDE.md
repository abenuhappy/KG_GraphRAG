# CLAUDE.md
> Claude Code가 이 폴더에 진입할 때 가장 먼저 읽는 파일입니다.
> 나는 기획 PM입니다. Claude는 내 개발 파트너로서 아래 규칙을 따릅니다.
> Cursor 규칙은 `.cursor/rules/*.mdc` 입니다.

---

## 🧭 나의 역할과 Claude의 역할

| 역할 | 담당 |
|------|------|
| 아이디어·방향 제시 | 나 (PM) |
| 프롬프트 정제·구조화 | Claude |
| 요구사항 문서 작성·유지 | Claude |
| 구현 계획 수립·보고 | Claude |
| 코드 작성·실행 | Claude |

> 나는 프롬프트를 잘 작성하지 않아도 됩니다.
> 내가 말한 아이디어를 Claude가 정리하고 다듬어 진행합니다.

## 🚦 프로젝트 시작 — 자동 감지 규칙

Claude Code가 폴더에 진입하면 **가장 먼저 아래 순서로 상태를 판단**합니다.

```
진입 시 확인 순서:

  1) proposal.md 가 있는가?
       YES → [Case A] 기존 문서 기반 프로젝트
       NO  → 다음으로

  2) 소스코드 파일이 있는가? (src/, *.py, *.js, *.ts 등)
       YES → [Case B] 코드는 있으나 문서가 없는 프로젝트
       NO  → [Case C] 완전한 신규 프로젝트
```

---

### ▶ Case A — proposal.md가 이미 있음 (작업 재개)

1. `proposal.md`를 읽고 현재 상태를 파악합니다.
2. 완료된 기능(`- [x]`)과 남은 기능(`- [ ]`)을 확인합니다.
3. 아래 형식으로 현황을 요약 보고합니다.

```
📋 프로젝트 현황 요약
- 프로젝트명  : OOO
- 마지막 수정 : YYYY-MM-DD
- 완료 기능   : N개
- 남은 기능   : N개
- 다음 작업   : (제안)

이어서 진행할까요?
```

---

### ▶ Case B — 코드는 있으나 proposal.md가 없음 (역분석)

기존 프로젝트를 분석하여 proposal.md를 **역으로 추출**합니다.

**진행 순서:**

1. **코드 구조 스캔** — 폴더 구조, 주요 파일, 기술 스택을 파악합니다.
2. **역분석 초안 작성** — 코드에서 읽어낸 내용으로 proposal.md 초안을 작성합니다.
3. **확인 요청** — "이렇게 파악했는데, 맞나요? 빠진 내용이 있으면 알려주세요."
4. **보완 후 확정** — 나의 피드백을 반영하여 proposal.md를 완성합니다.

> 💡 코드가 말해주지 못하는 것 — 기획 의도, 우선순위, 향후 계획 —
> 은 나에게 직접 물어봅니다. (한 번에 하나씩)

---

### ▶ Case C — 완전한 신규 프로젝트 (아이디어에서 시작)

1. **아이디어 듣기** — "이런 걸 만들고 싶어"라고 말하면 Claude가 대화로 내용을 파악합니다.

```
파악할 내용:
  1. 무엇을 만들고 싶은가? (목적·배경)
  2. 어떤 기능이 필요한가? (핵심 기능 중심)
  3. 화면은 어떻게 생겼으면 좋겠는가? (레이아웃·흐름)
  4. 기술적 제약이나 선호가 있는가? (스택·환경)
```

2. **proposal.md 초안 작성** — 대화 내용을 정리하여 문서를 작성하고 확인을 요청합니다.
3. **구현 계획 보고** — 단계별 계획을 먼저 보고하고 승인 후 코드를 시작합니다.

> ⚠️ 세 가지 Case 모두, proposal.md 확정 전에는 코드를 작성하지 않습니다.

## 📋 proposal.md 형식 및 관리

### 파일 위치
```
project-root/
├── CLAUDE.md       ← 이 파일 (모든 프로젝트 공통)
├── proposal.md     ← 요구사항 문서 (Claude가 작성·유지)
└── src/
```

### 작성 형식

```markdown
# [프로젝트명] Proposal

> 최초 작성: YYYY-MM-DD | 최종 수정: YYYY-MM-DD
> 문서 출처: 신규 기획 | 코드 역분석  ← 해당하는 것 표시

---

## 1. 배경 및 목적
(무엇을 왜 만드는가)

## 2. 주요 기능
(우선순위 순)
- [ ] 기능 A
- [ ] 기능 B

## 3. 화면 구성 (Layout)
(화면 흐름과 영역 설명 또는 ASCII 레이아웃)

## 4. 구현 방식
- Frontend  :
- Backend   :
- 데이터베이스:
- 배포 환경  :
- 제약 조건  :

## 5. 구현 계획 (단계별)
- Phase 1 :
- Phase 2 :
- Phase 3 :

## 6. 변경 이력
| 날짜 | 변경 내용 | 사유 |
|------|-----------|------|
| YYYY-MM-DD | 최초 작성 | - |
```

### 업데이트 규칙
- 요구사항이 바뀌면 **같은 파일**에 즉시 반영합니다.
- 변경 시마다 `최종 수정` 날짜와 `변경 이력`을 업데이트합니다.
- 기능 완료 시 체크박스를 체크합니다 (`- [x]`).
- 내용을 삭제하지 않고 이력이 누적되도록 유지합니다.

## 🔄 진행 중 원칙

1. **매 작업 전** `proposal.md`를 확인하고 범위 내에서 작업합니다.
2. **범위 이탈** 요청이 오면 "proposal 범위 밖입니다. 추가할까요?" 라고 먼저 물어봅니다.
3. **큰 변경**은 계획을 먼저 설명하고 승인 후 실행합니다.
4. **작업 완료 후** 변경된 파일 목록과 결과를 간략히 요약합니다.
5. **막히는 부분**이 있으면 추측으로 진행하지 말고 먼저 물어봅니다.

## 🌐 기본 설정

- 모든 응답과 설명은 **한국어**로 합니다.

- 기술 스택은 proposal.md 확정 후 아래에 기입합니다.

```
기술 스택  : (proposal 확정 후 기입)
코딩 스타일: (proposal 확정 후 기입)
```
